# Custom Metadata Feature

## Overview

The Custom Metadata feature allows you to define custom annotations in your OpenAPI specification that will be preserved and generated as Dart annotations in your model classes. This is useful for marking fields with special characteristics like auto-generated IDs, immutable fields, or any other domain-specific metadata.

## Configuration

### Basic Setup

Add the `custom_metadata` section to your `openapi_generator.yaml`:

```yaml
openapi_generator:
  schema_path: openapi.json
  output_directory: lib/generated
  json_serializer: freezed

  custom_metadata:
    enabled: true
    fields:
      - name: auto_generated
        annotation: '@AutoGenerated()'
      - name: immutable
        annotation: '@Immutable()'
      - name: deprecated_field
        annotation: '@Deprecated("Use alternative field")'
```

### Configuration Options

| Option | Type | Description |
|--------|------|-------------|
| `enabled` | boolean | Enable or disable custom metadata generation (default: `false`) |
| `fields` | array | List of custom metadata field definitions |

### Field Definition

Each field in the `fields` array has the following properties:

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `name` | string | Yes | The name of the metadata field in your OpenAPI spec |
| `annotation` | string | Yes | The Dart annotation to generate (e.g., `@MyAnnotation()`) |
| `annotation_template` | string | No | Template for annotations with values (use `{value}` placeholder) |

## Usage

### 1. Define Metadata in OpenAPI Specification

Add custom metadata fields to your property definitions in `openapi.json`:

```json
{
  "components": {
    "schemas": {
      "User": {
        "type": "object",
        "properties": {
          "id": {
            "type": "string",
            "description": "User ID",
            "auto_generated": true,
            "immutable": true
          },
          "email": {
            "type": "string",
            "description": "User email"
          },
          "created_at": {
            "type": "string",
            "format": "date-time",
            "auto_generated": true
          }
        }
      }
    }
  }
}
```

### 2. Define Annotation Classes

Create the annotation classes that correspond to your configured annotations:

```dart
// lib/annotations/field_metadata.dart

/// Marks a field as auto-generated by the backend
class AutoGenerated {
  const AutoGenerated();
}

/// Marks a field as immutable after creation
class Immutable {
  const Immutable();
}

const autoGenerated = AutoGenerated();
const immutable = Immutable();
```

### 3. Generate Code

Run the generator:

```bash
dart run openapi_retrofit_generator
dart run build_runner build -d
```

### 4. Generated Output

The generator will produce:

```dart
import 'package:freezed_annotation/freezed_annotation.dart';
import 'package:your_package/annotations/field_metadata.dart';

part 'user.freezed.dart';
part 'user.g.dart';

@Freezed()
abstract class User with _$User {
  const factory User({
    @autoGenerated @immutable required String id,
    required String email,
    @autoGenerated required DateTime createdAt,
  }) = _User;

  factory User.fromJson(Map<String, Object?> json) => _$UserFromJson(json);
}
```

## Examples

### Example 1: Basic Field Markers

**OpenAPI Specification:**
```json
{
  "id": {
    "type": "string",
    "auto_generated": true,
    "immutable": true
  }
}
```

**Configuration:**
```yaml
custom_metadata:
  enabled: true
  fields:
    - name: auto_generated
      annotation: '@autoGenerated'
    - name: immutable
      annotation: '@immutable'
```

**Generated Code:**
```dart
@autoGenerated @immutable required String id,
```

### Example 2: Using x- Prefix (OpenAPI Standard)

The generator supports both prefixed and non-prefixed field names:

**OpenAPI Specification:**
```json
{
  "id": {
    "type": "string",
    "x-auto-generated": true,
    "x-immutable": true
  }
}
```

**Configuration (same as above):**
```yaml
custom_metadata:
  enabled: true
  fields:
    - name: auto_generated  # Will match both 'auto_generated' and 'x-auto-generated'
      annotation: '@autoGenerated'
```

### Example 3: Template Annotations with Values

**OpenAPI Specification:**
```json
{
  "legacyField": {
    "type": "string",
    "deprecation_message": "Use newField instead"
  }
}
```

**Configuration:**
```yaml
custom_metadata:
  enabled: true
  fields:
    - name: deprecation_message
      annotation_template: '@Deprecated("{value}")'
```

**Generated Code:**
```dart
@Deprecated("Use newField instead") required String legacyField,
```

### Example 4: Database-Specific Annotations

**OpenAPI Specification:**
```json
{
  "properties": {
    "id": {
      "type": "string",
      "db_primary_key": true,
      "db_indexed": true
    },
    "email": {
      "type": "string",
      "db_indexed": true,
      "db_unique": true
    }
  }
}
```

**Configuration:**
```yaml
custom_metadata:
  enabled: true
  fields:
    - name: db_primary_key
      annotation: '@PrimaryKey()'
    - name: db_indexed
      annotation: '@Indexed()'
    - name: db_unique
      annotation: '@Unique()'
```

**Generated Code:**
```dart
@PrimaryKey() @Indexed() required String id,
@Indexed() @Unique() required String email,
```

## How It Works

### 1. Parsing Phase

During OpenAPI schema parsing, the generator:
- Checks if `custom_metadata.enabled` is `true`
- For each property definition, looks for fields matching the configured metadata field names
- Supports automatic detection of both `field_name` and `x-field_name` formats
- Stores the metadata values in `UniversalType` objects

### 2. Code Generation Phase

During code generation, the generator:
- Checks if any field has custom metadata
- Automatically adds necessary imports for annotation classes
- Generates annotations before field declarations in the correct order
- Handles both simple boolean flags and value-based metadata

### 3. Import Resolution

The generator automatically adds imports when custom metadata is detected. You can configure the import path by modifying the template or by ensuring your annotations are exported from your main package file.

## Best Practices

### 1. Use Descriptive Field Names

```yaml
# Good
fields:
  - name: auto_generated
  - name: immutable
  - name: db_indexed

# Less Clear
fields:
  - name: ag
  - name: imm
  - name: idx
```

### 2. Create a Dedicated Annotations File

Keep all your custom annotations in one place:

```dart
// lib/annotations/database_metadata.dart
class AutoGenerated {
  const AutoGenerated();
}

class Immutable {
  const Immutable();
}

class Indexed {
  const Indexed();
}

// Export constants for convenient usage
const autoGenerated = AutoGenerated();
const immutable = Immutable();
const indexed = Indexed();
```

### 3. Document Your Metadata Fields

Add comments to your OpenAPI schema explaining custom metadata:

```json
{
  "id": {
    "type": "string",
    "description": "Unique identifier",
    "auto_generated": true,  // Generated by backend, don't set manually
    "immutable": true        // Cannot be changed after creation
  }
}
```

### 4. Opt-In by Default

Keep the feature disabled by default and only enable it when needed:

```yaml
custom_metadata:
  enabled: false  # Explicit opt-out for teams that don't need it
```

### 5. Avoid Name Conflicts

If your annotation name conflicts with existing imports (e.g., `@immutable` from `package:meta`), use a unique name:

```yaml
fields:
  - name: immutable
    annotation: '@dbImmutable'  # Avoids conflict with meta package
```

## Limitations

### 1. Serializer Support

Currently, custom metadata annotations are only generated for the **Freezed** serializer. Support for `json_serializable` and `dart_mappable` can be added in future versions.

### 2. Annotation Classes Must Exist

The generator does not create annotation classes. You must define them in your project before running `build_runner`.

### 3. No Runtime Validation

The generated annotations are for documentation and tooling purposes. They don't automatically enforce constraints at runtime. You'll need to implement runtime logic separately.

### 4. Boolean and String Values Only

Currently optimized for boolean flags and string values. Complex object metadata may not work as expected with templates.

### 5. No Validation of Annotation Syntax

The generator does not validate that your annotation strings are valid Dart syntax. Invalid annotations will cause compilation errors.

## Troubleshooting

### Issue: "Undefined name used as an annotation"

**Cause:** Annotation class is not defined or not imported.

**Solution:**
1. Define the annotation class
2. Export it from your package
3. Ensure the generator adds the import (or add it manually)

### Issue: "Ambiguous import"

**Cause:** Annotation name conflicts with another import (e.g., `@immutable` from `package:meta`).

**Solution:** Use a unique annotation name in your configuration:

```yaml
fields:
  - name: immutable
    annotation: '@dbImmutable'  # Different name
```

### Issue: Metadata not appearing in generated code

**Cause:** Feature might be disabled or field names don't match.

**Solution:**
1. Check that `custom_metadata.enabled: true`
2. Verify field names match exactly (case-sensitive)
3. Check that the OpenAPI property has the metadata field

### Issue: Template values not substituting

**Cause:** Using `annotation` instead of `annotation_template`.

**Solution:** Use `annotation_template` with `{value}` placeholder:

```yaml
fields:
  - name: max_length
    annotation_template: '@MaxLength({value})'
```

## Migration Guide

### From Manual Annotations

If you were manually adding annotations after generation:

**Before:**
```dart
// Manually edited after generation
@autoGenerated required String id,
```

**After:**
1. Add metadata to OpenAPI spec
2. Configure custom_metadata
3. Regenerate - annotations appear automatically

### Adding to Existing Project

1. Add `custom_metadata` section to config (disabled)
2. Test generation - should work as before
3. Add metadata fields to OpenAPI spec
4. Create annotation classes
5. Enable feature and regenerate
